## Codebase Patterns
- This repository is Python-first; use `python -m mypy docos/` for type checking and `python -m pytest tests/ -q` for automated verification.
- Core runtime work is split across `docos/cli`, `docos/pipeline`, `docos/wiki`, `docos/lint`, `docos/harness`, and `docos/review`; keep each story focused on one boundary when possible.
- PipelineRunner (docos/pipeline/runner.py) is the unified entry point for full pipeline execution; new stages should be added as methods there.
- PIPELINE_STAGES in docos/models/run.py defines canonical stage order: ingest, route, parse, normalize, extract, compile, patch, lint, harness, gate.
- Config for tests should use inline YAML strings to avoid external file dependencies.
- Only stdlib_pdf and basic_text parsers are available in the registry; router.yaml references external parsers (pymupdf etc.) that are NOT registered.
- ParserRouter now accepts an optional `parser_registry` parameter in __init__ for validate_config(); existing code that doesn't pass it continues to work.
- PDF dual-column detection works by analyzing x-coordinate clusters from Td positioning commands in content streams.
